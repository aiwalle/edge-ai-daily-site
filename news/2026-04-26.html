端智能日报 2026-04-26

本期聚焦：快手 OneRec 生成式推荐系统披露完整工程数据，AIGC 素材驱动 Q4 广告营收实打实增长 5%，是目前端智能落地广告商业化最具说服力的大厂案例；同期 Meta 发布 RPRA 端云路由新范式，小模型在端侧预测"会不会答好"再决定是否上云，为广告链路的端云协同决策提供新思路。

1. 【快手 OneRec：生成式推荐承接 25% 线上流量，AIGC 广告素材驱动 Q4 营收 +5%】

  • 评分：A 5/5 · I 4/5 · S=14 · P0 · 价值类型：收入提升 / 成本优化
  • 来源：快手技术团队 | arxiv.org/abs/2506.13695 · 财报分析（新浪财经） | 论文 2025-06，财报分析 2026-04-09

内容总结（150 字）： 快手端到端生成式推荐系统 OneRec 最新披露运营数据：已在快手 / 快手极速版双端承接 25% 线上流量，APP 停留时长分别提升 0.54% / 1.24%；本地生活场景 AB 实验 GMV 提升 21.01%，订单量 +17.89%，新客获取效率 +23.02%。工程侧：通过算子压缩（1,200 个算子）将训练 / 推理 MFU 提升至 23.7% / 28.6%，通信与存储运营成本削减近 90%，在推荐模型 FLOPs 提升 10× 的同时实现降本。广告侧：AIGC 营销素材 Q4 2025 带动消耗超 40 亿元，生成式推荐大模型 OneRec + 智能出价模型协同推动国内线上营销服务收入提升约 5%。2026 年计划 Capex 约 260 亿元，其中 110 亿净增量主要用于 Keling 等基础大模型算力。

业务价值与适配性： 可以让我们在音频/内容流媒体平台参考快手路径——用端到端生成式推荐替代多级级联漏斗，将召回、粗排、精排合并为单一自回归模型，直接生成候选 item 的语义 ID，绕开传统 CTR 模型的 feature engineering 瓶颈。可借鉴思路：① OneRec 的"协同感知多模态分词器 + RQ-Kmeans 分层语义 ID"方案可迁移到广告素材理解；② AIGC 素材 + 生成式推荐联动的"内容-分发"一体化飞轮是近期最值得立项调研的方向；③ MFU 28.6% 的推理效率说明超大规模生成式推荐已具备生产部署可行性。

2. 【RPRA（Meta，arxiv 2604.12634）：端侧小模型预测 LLM-Judge 分数实现端云推理路由】

  • 评分：A 4/5 · I 3/5 · S=11 · P1 · 价值类型：成本优化 / 端云协同
  • 来源：Meta Platforms / USI | arxiv.org/abs/2604.12634 | 2026-04-14

内容总结（140 字）： Meta 提出 RPRA（Reason-Predict-Reason-Answer） 端云路由范式：端侧小模型在生成回答前，先预测大模型 Judge 会给自己打几分，若预测分数达标则直接在端侧作答，否则将请求路由至云端大模型。核心贡献：① 形式化 PA（Predict-Answer）和 RPRA 两种范式；② 提出"成绩单（Report Card）"——跨数据集汇总小模型表现，无需逐条调用 Judge；③ 用 Hindsight Trick 做监督微调，绕过推理时 Judge 调用成本。实测模型：MobileLLM 0.9B、Llama 3.2 1B/3B、Llama 3.1 8B。三种实现路径：零样本预测、In-context Report Card、监督微调，其中微调后的 Llama 3.2 3B 路由准确率最高。

业务价值与适配性： 可以让我们在广告相关性 / 素材质量评估链路中用端侧小模型（如 MobileLLM 0.9B）做"能不能本地判断"的门控，只在本地模型不确定时才上云调用大模型 Judge，直接削减云端推理成本。可借鉴思路：将 RPRA 的路由逻辑用于广告意图识别场景——端侧先判"意图是否明确"，明确则本地输出，模糊则上云，形成低延迟 + 低成本的端云协同决策链路；"Report Card"机制也可用于监控端侧模型在各广告场景的置信度分布。

3. 【XPerT（MobiSys 2025，arxiv 2504.13938）：端侧 LLM 个性化加速——模型选择节省 83% 计算、51% 数据】

  • 评分：A 4/5 · I 3/5 · S=11 · P1 · 价值类型：用户体验 / 成本优化
  • 来源：U. Pittsburgh / Google | arxiv.org/abs/2504.13938 | MobiSys 2025（2025-06），arxiv 2025-04-15

内容总结（140 字）： XPerT（eXplainable Personalized Tuning）解决端侧 LLM 个性化的冷启动问题：服务端预先缓存大量已针对不同用户群微调的 pLLM（Personalized LLM），用户设备只需从中选择最合适的一个再做少量本地微调，而非从基础模型从头 fine-tune。选择机制：服务端为每个 pLLM 计算"与基础模型的输出分布漂移"作为 explanation，传给端侧；端侧将 explanation 与本地行为数据对比，无需实际运行每个候选模型即可完成选择（所有 pLLM 共享一组 sub-vectors，选择开销比现有方法低 96.5%）。关键指标：在多款 Android 手机实测，选择准确率 96%，相比从头 fine-tune 节省 83% 墙钟计算时间，数据效率提升 51%。

业务价值与适配性： 可以让我们在广告用户建模场景下，为不同用户群（高价值用户、新用户、特定兴趣用户）维护一批预微调的端侧推荐/意图模型，按需下发"最匹配的那个"而非统一模型，实现隐私保护的个性化。可借鉴思路：XPerT 的"服务端缓存 pLLM + 端侧轻量选择"架构可直接迁移到广告 SDK 的个性化模型分发——结合 A/B 实验框架，将不同用户段的模型漂移值作为调度依据，兼顾效果与隐私。

4. 【Vikas Chandra（Meta AI Research）：On-Device LLMs State of the Union 2026——30-50× 带宽差距与工程应对全景】

  • 评分：A 3/5 · I 4/5 · S=10 · P2 · 价值类型：新能力探索
  • 来源：Meta AI Research（Vikas Chandra & Raghuraman Krishnamoorthi）| v-chandra.github.io/on-device-llms | 2026 年初

内容总结（130 字）： Meta AI Research 对端侧 LLM 工程现状的系统性梳理，核心数字：① 内存带宽瓶颈——移动端 50-90 GB/s vs 数据中心 GPU 2-3 TB/s，差距 30-50×，decode 阶段每生成一个 token 需要将全部权重流过内存，INT4 量化直接将内存流量压缩 4×，是提升 token/s 最直接的手段；② NPU 算力——旗舰移动 NPU 已接近 V100 的 125 TOPS，prefill 阶段可充分利用；③ 投机解码——Medusa 加速 2.2-3.6×、EAGLE 同类加速并具备更高接受率；④ 框架矩阵：ExecuTorch（50KB 运行时，12+ 后端）、llama.cpp（CPU 原型）、MLX（Apple Silicon）；⑤ 推荐模型尺寸：Llama 3.2 1B/3B、Gemma 3 270M、Phi-4 mini 3.8B、Qwen2.5 0.5B-1.5B。

业务价值与适配性： 可以让我们在选型端侧广告模型时以"内存带宽 / 模型大小 × 量化位数"为核心约束而非 TOPS，重新校准模型压缩优先级——INT4 比 INT8 更关键，投机解码在广告 CTR 快速响应场景有直接价值。可借鉴思路：用该文的带宽分析框架做端侧广告模型的 latency 预估（不依赖实机测试），帮助在立项阶段快速筛选可行参数量范围。

5. 【SqueezeBits Yetter：Apple Silicon NPU Prefill + GPU Decode 异构分离推理】

  • 评分：A 2/5 · I 3/5 · S=7 · P2 · 价值类型：成本优化 / 新能力探索
  • 来源：SqueezeBits Tech Blog | blog.squeezebits.com/disaggregated-inference-on-apple-silicon | 近期发布

内容总结（120 字）： SqueezeBits 推出 Yetter Inference Engine，在 Apple Silicon 上实现 prefill/decode 异构分离：CoreML + ANE（Neural Engine）负责 prefill（固定 shape 输入、批量并行，ANE 效率最优），MLX + GPU 负责 decode（逐 token 自回归，MLX 延迟更低）。两阶段通过 iOS 18 / macOS 15 引入的 Stateful Models 共享 KV cache，单一模型文件包含 prefill 和 decode 两个函数，权重共享保持文件体积不增。工程意义：充分利用 Apple Silicon 统一内存架构，两个计算单元并行流水线，避免 CoreML 在 decode 阶段的延迟劣势和 MLX 在长 context prefill 的能效劣势。

业务价值与适配性： 可以让我们在 iOS 广告 SDK 中以更低延迟处理长用户上下文输入（prefill 用 NPU）同时保持快速 token 生成（decode 用 GPU），对"用户历史行为序列 → 广告意图推理"场景直接有收益。暂不适用：Yetter Engine 目前为闭源商业工具，需评估接入成本；但其 Stateful Models + 函数分离的思路可在自有 CoreML pipeline 中复现。
