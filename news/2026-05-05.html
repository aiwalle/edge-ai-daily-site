端智能日报 2026-05-05

本期聚焦：端侧推荐/广告场景迎来两项有实测数据的工程成果——OD-LLM（WSDM 2026）首次为序列推荐 LLM 构建任务自适应压缩框架，模型减半精度零损失；EdgeWisePersona 提供首个用于评估端侧 SLM 用户画像能力的公开基准；TurboQuant 的 3-bit KV Cache 压缩方案（ICLR 2026）正被社区快速移植到 llama.cpp，Q2 2026 量产路径明确——三项合力标志着端侧广告推理的"精度-内存-成本"三角已出现系统性破局点。

1. 【OD-LLM（WSDM 2026）：序列推荐 LLM 的端侧任务自适应压缩——SVD 低秩分解 + 渐进对齐，模型减半 · 精度零损失，已在 WSDM 2026 发表】

  • 评分：A/5 · I/3 · S=13 · P0 · 价值类型：收入提升/成本优化
  • 来源：多机构 | arxiv.org/abs/2601.09306 | WSDM 2026（2 月 22–26 日，Boise）

内容总结（130 字）： 现有端侧 LLM 推荐方案直接借用通用模型压缩技术，忽视了序列推荐任务的特殊性：token 分布与 NLP 任务差异巨大，导致激进压缩时精度急剧下降。OD-LLM 是首个任务自适应压缩框架，三步解决：① 低秩结构压缩（SVD 分解）大幅削减参数冗余；② Tokenization 归一化，对齐推荐 token 分布与 SVD 分解的统计假设；③ 渐进对齐算法，逐层迭代精化目标模型参数，防止高压缩比时误差累积。在多个序列推荐 benchmark 上验证：压缩至原始模型 50% 参数量，精度无损；高压缩比下仍显著优于通用方法（LLM-Pruner、LoRA-based 方案等）。端侧部署目标：4GB RAM 以内旗舰手机实时序列推荐。

业务价值与适配性： 可以让我们在广告序列推荐（Session-based CTR/CVR）场景中把 LLM-based 召回/排序模型直接部署到端侧，推理无需云端往返，延迟从 200ms+ 降至 <20ms。可直接应用：SVD + 渐进对齐流程可独立于模型架构使用，建议以内部序列推荐 LLM（DIN/DIEN 扩展版）为基线，2 周内跑 50% 压缩比的精度对齐实验，评估 AUC 损失是否在 0.2% 以内。

2. 【EdgeWisePersona（arXiv 2505.11417，TCL Research Europe）：首个端侧 SLM 用户画像评测基准——多轮对话行为重建 · 隐私保护 · 开源数据集 + 代码】

  • 评分：A/4 · I/3 · S=11 · P1 · 价值类型：新能力探索/用户体验
  • 来源：TCL Research Europe | arxiv.org/abs/2505.11417 | 2026-05

内容总结（130 字）： 在广告/推荐系统中，用户画像（User Profiling）是精准定向的核心，而端侧本地化画像兼顾隐私合规与实时个性化。然而业界长期缺乏系统性评测基准。EdgeWisePersona 填补此空白：① 数据集构造：以智能家居场景为载体，用 LLM 生成基于"行为惯例（Routine）"的多轮自然语言对话，形成"用户偏好结构 → 对话历史"配对；② 评测任务：Profile Reconstruction（从多轮交互历史推断用户惯例与偏好）——天然映射广告系统的"用户兴趣建模"；③ 基准对象：专为资源受限边缘设备设计的 SLM（Small Language Model），强调隐私保护与可解释性。数据集和评测代码已开源（TCLResearchEurope/EdgeWisePersona，Hugging Face + GitHub）。端侧延迟/内存：依具体 SLM 而定，本文聚焦 benchmark 构建，配套实验模型为 1–3B 量级。

业务价值与适配性： 可以让我们在广告 SDK 的端侧用户画像模块引入系统性评测流程——当前业界缺乏标准化 benchmark，导致端侧画像精度难以横向对比。可借鉴思路：EdgeWisePersona 的 Profile Reconstruction 评测范式可直接迁移到广告场景：将用户广告点击/时长/跳出历史替换对话历史，验证端侧 SLM 推断兴趣类目的准确率；评估完成后可接入 OD-LLM（条目 1）压缩的模型作为端侧画像引擎，形成完整链路。

3. 【Google TurboQuant（ICLR 2026）社区实现进展：llama.cpp PR #21089 开放 · TheTom/turboquant_plus 6.4k stars · 3-bit KV Cache 5.2x 压缩即将落地移动端】

  • 评分：A/3 · I/4 · S=10 · P1 · 价值类型：成本优化/用户体验
  • 来源：Google DeepMind + NYU（Zandieh et al.）| ICLR 2026 | 社区实现：github.com/ggml-org/llama.cpp/discussions/20969

内容总结（130 字）： TurboQuant（ICLR 2026）是 Google DeepMind + NYU 提出的 KV Cache 极限量化方案：① 随机正交旋转将 KV 向量能量均匀分布，使量化误差在统计上最优；② Lloyd-Max 最优码本预计算，无需校准数据，任意模型/层/head 通用；③ 额外引入 1-bit 残差（Quantized Johnson-Lindenstrauss 变换）保持向量距离关系。结果：3.5-bit 压缩比 FP16 内存减少 5.2x，质量几乎无损（Llama-3.1-8B、Mistral-7B recall 与 FP16 一致）；H100 上 4-bit TQ 加速注意力计算 8x。社区落地：llama.cpp PR #21089（OPEN）实现 tbq3_0/tbq4_0 两种新 cache 类型；TheTom/turboquant_plus fork（6.4k stars）已先于主线上线；官方 Q2 2026 发布预期。

业务价值与适配性： 可以让我们在广告端侧推理中将 KV Cache 内存占用降低 5x——这对于上下文较长的广告序列推理（用户历史行为序列 512-2048 token）尤为关键，直接决定旗舰机能否流畅运行 7B 级模型而不触发热降频。可借鉴思路：待 PR #21089 合入主线后（预计 Q2 2026），建议立即在广告 SDK 内部的 llama.cpp 分支中启用 tbq4_0，与 arXiv 2603.23640（持续推理热降频研究，05-04 报道）配合使用，可显著延长 NPU-first 路径下的持续推理窗口。

4. 【Qwen3.6-35B-A3B 开源发布（阿里 Qwen 团队，Apache 2.0）：35B 总参数 · 3B 推理激活 · 多模态 MoE · 73.4% SWE-Bench，高吞吐低成本端云推理新选项】

  • 评分：A/3 · I/4 · S=10 · P1 · 价值类型：新能力探索/成本优化
  • 来源：阿里 Qwen 团队 | qwen.ai/blog?id=qwen3.6-35b-a3b | MarkTechPost 报道 | 2026-04-16

内容总结（130 字）： Qwen3.6-35B-A3B 是阿里巴巴 Qwen 团队以 Apache 2.0 开源的稀疏 MoE 视觉语言模型：总参数 35B，推理时仅激活 3B 参数，每 token 算力成本接近 3B 密集模型；同时支持图像、视频、文档理解。核心指标：SWE-Bench 73.4%（代理编程任务），超越同量级密集模型；262K token 上下文窗口。推理框架：支持 vLLM、SGLang、KTransformers、llama.cpp（GGUF 版本由 unsloth 发布）；FP8 量化后约 35GB，可装入单张 H100。端侧定位：总权重体积（FP16 ~70GB）决定其不适合直接手机部署，但 3B 激活量意味着在服务端可以超低成本跑广告大模型，是广告云侧智能体的高性价比选择；配合未来手机内存扩展，边缘推理有探索空间。

业务价值与适配性： 可以让我们在广告素材理解（图文/视频多模态）场景中以等效 3B 的计算成本运行能力接近 27B 密集模型的多模态 LLM，大幅降低广告审核/素材描述生成的云端推理成本。可直接应用（云侧）：GGUF 版本已在 unsloth 发布，建议 1 周内部署 Q4 量化版本（约 17-20GB），对比内部当前多模态素材理解模型的精度-延迟-成本三角；暂不适用（端侧）：需等待更小参数量的 Qwen3.6 系列（预计有 1.5B/3B dense 版本）面世。
