端智能日报 2026-04-20

本期聚焦：Google 三线发力端侧 LLM 基础设施（Gemma 4 + LiteRT-LM + Firebase 混合推理 API）、Meta 三项端侧工程更新（ExecuTorch v1.2.0 / EUPE 视觉编码器 / State of Union 报告）、端侧 CTR 与用户建模论文、Qualcomm 硬件趋势。

1. 【Google 发布 LiteRT-LM：端侧 LLM 生产级推理框架正式开源】

  • 来源：Google Developers Blog（原文）| GitHub | 2026-04-07
  • 摘要：LiteRT-LM 是 TFLite 的 LLM 继任者，支持 Android/iOS/Web/桌面/IoT，内存占用低至 1.5 GB，延迟 <100ms，已在 Chrome、Pixel Watch、Chromebook Plus 量产落地；支持 Gemma/Llama/Phi-4/Qwen，提供 GPU/NPU 硬件加速与结构化输出。
  • 值得关注的原因：这是 Google 将端侧 LLM 推理能力产品化的里程碑，直接替代 TFLite 成为 Android 端模型部署新标准；流媒体 App 若要做端侧 CTR 预估或意图识别，LiteRT-LM 是优先评估的运行时框架。

2. 【Google Gemma 4 E2B/E4B：首个支持视觉+音频+函数调用的端侧多模态开源模型】

  • 来源：Google DeepMind（官方页面）| Edge AI & Vision Alliance 解读 | 2026-04-02
  • 摘要：Gemma 4 Edge 系列（E2B/E4B）支持图像、音频、文本多模态输入及结构化函数调用，E2B 仅需 1.5 GB 内存，在 Qualcomm IQ8 NPU 上达 31 token/s；通过 per-layer embedding 设计显著提升参数效率；Apache 2.0 可商用。
  • 值得关注的原因：音频理解 + 函数调用 + 端侧运行直接对应内容流媒体平台的用户意图理解场景；E2B 内存占用已达旗舰机可接受范围，可作为端侧广告意图识别模型的技术选型参考。

3. 【Meta ExecuTorch v1.2.0：新增实时语音推理与 Qwen3.5 导出支持】

  • 来源：ExecuTorch GitHub Releases | Arm Newsroom | 2026-04-01
  • 摘要：ExecuTorch v1.2.0 新增 Voxtral Realtime（流式语音推理）、NVIDIA NeMo Sortformer（说话人分离）、Silero VAD，支持 Qwen3.5 与 Parakeet 导出；Cortex-M 晋升为一级嵌入式目标；二进制体积进一步压缩；已在 Instagram、WhatsApp、Facebook 亿级规模生产部署。
  • 值得关注的原因：支持 12+ 硬件后端（Apple/Qualcomm/Arm/MediaTek/Vulkan），是目前覆盖最广的跨平台端侧推理框架；实时语音能力对内容 App 的语音广告理解场景有直接价值。

4. 【Meta 发布 EUPE：86M 参数视觉编码器通过三阶段蒸馏超越专家模型】

  • 来源：MarkTechPost（报道）| arXiv 2603.22387 | 2026-04-06
  • 摘要：EUPE（Efficient Universal Perception Encoder）通过三阶段蒸馏将 PEcore-G、PElang-G、DINOv3-H+ 三类专家大模型知识压缩至 86M ViT-B，在图像理解（IN1k-KNN 84.1）、密集预测（ADE20k 52.4 mIoU）、VLM（RealworldQA 55.5）三类任务上均超越同规模专家模型；iPhone 15 Pro 上 ViT-T 仅 6.8ms（ExecuTorch）。
  • 值得关注的原因：三阶段蒸馏方法论可迁移至广告模型压缩流程；对广告素材理解（封面 CTR 视觉特征提取）有直接工程参考意义。

5. 【arxiv 2602.13971 DAIAN：触发式推荐场景的意图感知 CTR 预估网络】

  • 来源：arXiv（2602.13971）| 2026-02
  • 摘要：DAIAN（Deep Adaptive Intent-Aware Network）将用户意图建模与 CTR 预估融合，通过分析用户点击 item 与触发 item 的相关性提取个性化意图表征，在触发式推荐场景下显著提升排序效果，支持端侧轻量化部署。
  • 值得关注的原因：触发式推荐与移动端广告重排逻辑高度相似（触发 item 即广告上下文），DAIAN 的意图感知设计可直接迁移到广告 rerank 模块，是本期最具业务参考价值的论文。

6. 【arxiv 2602.06622 R2LED：语义 ID 赋能终身用户行为序列 CTR 预估】

  • 来源：arXiv（2602.06622）| 2026-02
  • 摘要：R2LED 在终身用户行为序列建模中引入语义 ID（Semantic IDs），结合检索（Retrieval）和精化（Refinement）两阶段设计，解决长序列 CTR 预估中的检索效率与精度问题，冷启动场景优于传统 ID embedding。
  • 值得关注的原因：长序列用户建模是移动端广告的核心挑战；Retrieval+Refinement 两阶段设计天然适合端云协同——端侧轻量精化、云侧语义检索，降低存储与计算成本。

7. 【arxiv 2604.04988：Prune→INT8 QAT→蒸馏有序压缩流水线实现最优延迟-精度权衡】

  • 来源：arXiv（2604.04988）| 2026-04-05
  • 摘要：论文提出以"实测延迟"为优化目标的压缩流水线：先做无结构剪枝降低容量，再做 INT8 量化感知训练（主导运行时收益），最后用知识蒸馏恢复精度；实验验证该顺序在延迟-精度曲线上优于其他排列组合。
  • 值得关注的原因：直接给出"工程实用"的压缩顺序结论，可作为对广告重排模型进行端侧压缩时的标准化 SOP；与 ICLR 2026 同期论文（2603.18426）形成互补，值得一起读。

8. 【arxiv 2601.22543 SCaLRec：云侧语义缓存+端侧校准解决端侧重排语义过期问题】

  • 来源：arXiv（2601.22543）| 2026-01-30
  • 摘要：SCaLRec 提出云端协同推荐框架：云侧大模型生成用户语义向量并缓存到设备，设备端轻量校准模块在每次请求时利用最新交互修正缓存向量，无需每次调用云侧 LLM，在多个真实数据集上显著优于缓存复用基线。
  • 值得关注的原因：精准命中"端侧重排 + 用户建模"核心痛点——云侧语义过期；SCaLRec 的校准机制可直接借鉴，用极低额外开销保持语义向量时效性。

9. 【Google Firebase 推出混合推理 API：端侧与云端 Gemini 统一路由，应用无缝切换】

  • 来源：Android Developers Blog（原文）| 2026-04-17
  • 摘要：Firebase AI Logic 新增实验性混合推理 API，通过简单规则路由（PREFER_ON_DEVICE / PREFER_IN_CLOUD）在设备端 Gemini Nano 与云端 Gemini 之间动态切换，统一 API 封装屏蔽底层差异；端侧当前支持单轮文本/图像输入，云端可调用完整 Gemini 能力；Google 同期发布 Nano Banana 图像生成模型供 Android 应用调用。
  • 值得关注的原因：端云协同推理的工程接口正在标准化——对移动广告链路直接可用：实时竞价（低延迟，端侧优先）与复杂内容理解（精度优先，云侧兜底）可通过同一 API 按策略路由，降低多推理路径维护成本。

10. 【Qualcomm：Edge AI 进入"混合智能"工程化阶段，NPU 已成手机标配】

  • 来源：Edge AI San Diego 2026 大会 / Qualcomm 主题演讲（会议摘要）| 2026-04-07
  • 摘要：Qualcomm EVP Durga Malladi 指出端侧 AI 正从"能否运行"转向"如何工程化"，核心趋势是芯片+OS+云的混合智能架构，sub-10B 参数专属微调模型在垂直场景超越大型云端模型，Snapdragon NPU 已在主流旗舰机全面普及。
  • 值得关注的原因：为端侧广告算法选型提供硬件基准——旗舰 Android 设备已具备 NPU 加速能力，端侧 CTR 预估的延迟优化策略应优先考虑 NPU delegate 路径（LiteRT-LM 已原生支持）。
